# جلسه‌ی ۱۶ — تست‌نویسی 🧪

**هدف (۱ ساعت):** تست داخلی Go را یاد بگیری — به فریم‌ورک شخص‌ثالث نیازی نیست. تست
واحد، سبک اصطلاحی **جدول‌محور (table-driven)**، زیرتست‌ها و بنچمارک می‌نویسی و
**پوشش (coverage)** را اندازه می‌گیری. تست مهارتی است که با صدای بلند به کارفرما
«توسعه‌دهنده‌ی حرفه‌ای» را نشان می‌دهد، و هر کدبیس جدی Go تست دارد.

> **مرور جلسه‌ی ۱۵:** داده را با JSON و فایل جابه‌جا می‌کنی. حالا یاد می‌گیری *اثبات
> کنی کدت کار می‌کند* — و حین تغییر، کارکردنش را حفظ کنی.

---

## ۱. اولین تست تو (۱۵ دقیقه)

تست Go بخشی از زنجیره‌ابزار است. قواعد:

- فایل‌های تست به **`_test.go`** ختم می‌شوند و **کنار** کدی که تست می‌کنند هستند.
- توابع تست با **`Test`** شروع می‌شوند، حرف‌بزرگ‌اند و **`*testing.T`** می‌گیرند.
- با **`go test`** اجرایشان می‌کنی.

با تابعی در `calc.go`:

</div>

```go
func Add(a, b int) int { return a + b }
```

<div dir="rtl">

تست در `calc_test.go`:

</div>

```go
package calc

import "testing"

func TestAdd(t *testing.T) {
    got := Add(2, 3)
    want := 5
    if got != want {
        t.Errorf("Add(2, 3) = %d; want %d", got, want)
    }
}
```

<div dir="rtl">

الگو: `got` را حساب کن، با `want` مقایسه کن، و وقتی فرق دارند `t.Errorf` (با پیام
واضح) صدا بزن. هیچ کلیدواژه‌ی `assert` نیست — از `if` ساده استفاده می‌کنی.

| متد | اثر |
|-----|-----|
| `t.Errorf(...)` | تست را شکست‌خورده علامت بزن، **ادامه بده** |
| `t.Fatalf(...)` | شکست‌خورده علامت بزن و **همین تست را فوراً متوقف کن** (وقتی ادامه بی‌فایده است، مثلاً بعد از خطای غیرمنتظره) |

اجرایش کن:

</div>

```bash
go test ./examples/session16/calc/
go test -v ./examples/session16/calc/   # verbose: هر تست را نشان می‌دهد
```

## ۲. تست‌های جدول‌محور — اصطلاح Go (۲۰ دقیقه)

به‌جای نوشتن یک تابع جدا برای هر حالت، توسعه‌دهنده‌های Go حالت‌ها را در یک
**اسلایس از استراکت** می‌گذارند و رویشان حلقه می‌زنند. این رایج‌ترین الگوی تست در کد واقعی Go است:

</div>

```go
func TestIsEven(t *testing.T) {
    cases := []struct {
        name string
        in   int
        want bool
    }{
        {"zero is even", 0, true},
        {"three is odd", 3, false},
        {"negative even", -4, true},
    }

    for _, c := range cases {
        t.Run(c.name, func(t *testing.T) {   // یک زیرتست نام‌دار برای هر حالت
            if got := IsEven(c.in); got != c.want {
                t.Errorf("IsEven(%d) = %v; want %v", c.in, got, c.want)
            }
        })
    }
}
```

<div dir="rtl">

چرا محبوب است:
- **افزودن حالت = افزودن یک خط.** گسترش آسان.
- **`t.Run` زیرتست می‌سازد** — شکست‌ها حالت دقیق را نام می‌برند (`TestIsEven/three_is_odd`).
- مثل یک **مشخصات** رفتار تابع خوانده می‌شود.

> 🔑 این سبک جدول‌محور + `t.Run` همانی است که مصاحبه‌گرها انتظار دیدنش را دارند.
> پیش‌فرضت کن.

### تست خروجی خطا

برای الگوی `value, err`، هم مسیر خوش و هم مسیر شکست را تست کن — و برای بررسی یک
خطای مشخص از `errors.Is` (جلسه‌ی ۱۱) استفاده کن:

</div>

```go
_, err := Divide(10, 0)
if !errors.Is(err, ErrDivideByZero) {
    t.Errorf("expected ErrDivideByZero, got %v", err)
}
```

<div dir="rtl">

پوشه‌های [`examples/session16/calc/`](../examples/session16/calc/) و
[`examples/session16/stringutil/`](../examples/session16/stringutil/) را اجرا کن:

</div>

```bash
go test -v ./examples/session16/...
```

## ۳. پوشش — چقدر تست شده؟ (۱۰ دقیقه)

Go اندازه می‌گیرد چه کسری از دستورات کدت را تست‌هایت تمرین می‌دهند:

</div>

```bash
go test -cover ./examples/session16/...
# ok  golearn/examples/session16/calc  coverage: 100.0% of statements
```

<div dir="rtl">

برای گزارش بصری، یک نمای HTML بساز و باز کن:

</div>

```bash
go test -coverprofile=cover.out ./examples/session16/calc/
go tool cover -html=cover.out      # مرورگری که خطوط تست‌شده را هایلایت می‌کند باز می‌شود
```

<div dir="rtl">

> 💡 **پوشش یک راهنماست، نه یک هدف.** پوشش ۱۰۰٪ یعنی بدون باگ نیست، و دنبال کردن
> ۱۰۰٪ همه‌جا وقت هدر می‌دهد. هدف بگذار منطق مهم و حالت‌های لبه‌ی دشوار را پوشش
> دهی. کارفرماها عاشق دیدن تست‌های *معنادار*اند، نه فقط یک عدد.

## ۴. بنچمارک‌ها (۱۰ دقیقه)

بنچمارک‌ها کارایی را اندازه می‌گیرند. با **`Benchmark`** شروع می‌شوند،
**`*testing.B`** می‌گیرند و **`b.N`** بار حلقه می‌زنند (Go مقدار `b.N` را برای
اندازه‌گیری پایدار انتخاب می‌کند):

</div>

```go
func BenchmarkAdd(b *testing.B) {
    for i := 0; i < b.N; i++ {
        Add(100, 200)
    }
}
```

```bash
go test -bench=. ./examples/session16/calc/
go test -bench=. -benchmem ./examples/session16/calc/   # تخصیص حافظه را هم گزارش می‌دهد
```

<div dir="rtl">

خروجی نانوثانیه بر عملیات (`ns/op`) و با `-benchmem` بایت و تعداد تخصیص بر عملیات
را نشان می‌دهد — هنگام بهینه‌سازی بسیار ارزشمند است.

> 📦 **افزودنی‌های مفید تست:** `go test ./...` همه‌ی تست‌های پروژه را اجرا می‌کند.
> `t.Helper()` یک تابع را کمک‌تست علامت می‌زند (تا شکست‌ها به فراخواننده اشاره
> کنند). برای پروژه‌های بزرگ‌تر جامعه از `testify` برای assertهای بهتر استفاده
> می‌کند، اما کتابخانه‌ی استاندارد کافی است — و مصاحبه‌گرها احترام می‌گذارند که راه
> داخلی را بلدی.

## 🎯 تمرین‌ها (قبل از جلسه‌ی ۱۷ انجام بده!)

یک پکیج `examples/session16/practice/` با کد **و** یک `_test.go` بساز:

۱. **تست یک تابع:** `Max(a, b int) int` و یک `TestMax` که a>b، b>a و a==b را پوشش دهد.
۲. **جدول‌محور:** `Clamp(n, lo, hi int) int` (n را در [lo,hi] محدود می‌کند) و تست جدول‌محور با حداقل ۵ حالت، هرکدام یک زیرتست نام‌دار.
۳. **مسیر خطا:** `ParsePositive(s string) (int, error)` که یک int تجزیه می‌کند و اگر ≤ ۰ بود خطا می‌دهد. حالت موفق و هر دو حالت شکست (غیرعددی، و صفر/منفی) را تست کن.
۴. **اجرای پوشش:** `go test -cover` را روی پکیجت اجرا کن و سعی کن به پوشش بالا و معنادار برسی.
۵. **بنچمارک:** `Fib(n int) int` (فیبوناچی بازگشتی) و یک `BenchmarkFib` بنویس. اجرایش کن و `ns/op` را بخوان.
